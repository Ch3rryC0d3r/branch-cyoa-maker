Introduction

Branch is a lightweight engine and editor for creating Choose Your Own Adventure (CYOA) style stories. It‚Äôs designed for writers and devs who want to build branching narratives without drowning in complex scripting or custom engines.

Why Use Branch?

  • Fast Iteration: Test story changes instantly in Play Mode.
  • Simple Syntax: Write your story using a plain text format that‚Äôs easy to read and edit.
  • Expandable: Conditions, actions, timers, and variables make it powerful enough for complex narratives.
  • Portable: Branch projects are saved and natively loaded with plain JSON. JSON files are also really easy to share with people (it's basically a text file!)

Who is it For?

Branch is useful if you‚Äôre:

  • A writer who wants to prototype interactive stories without coding knowledge.
  • A game dev who needs a flexible branching dialogue/quest system.
  • A teacher creating quizzes or interactive lessons.

Basic Workflow

The storytelling flow in Branch looks like this:

Nodes ‚Üí contain Leaves (choices) ‚Üí each Leaf may have Conditions + Actions ‚Üí lead to the next Node

With this flow, you can build small branching adventures or large-scale interactive narratives.

Core Concepts

Everything in Branch revolves around a few simple but powerful concepts. Once you understand these, you‚Äôll have the tools to build any branching narrative you can imagine.

Nodes

Nodes are the building blocks of your story. Each node represents a point in the narrative where the player reads text and makes choices.

Example Node:

[1] The cave entrance is dark and cold.
  Take the torch | 2 | | add_item:Torch
  Step inside | 3 | has_item:Torch |

Leaves

Leaves are the choices inside each node. They handle branching logic and trigger actions. (See the detailed Leaves section for a deep dive.)

Conditions

Conditions control visibility of leaves and sometimes nodes themselves. They let you write logic like:

  • ‚ÄúThis option only appears if the player has a key.‚Äù
  • ‚ÄúThis fight option appears if HP > 0.‚Äù

Actions

Actions are commands that fire when a player chooses a leaf or enters a node. They can change variables, inventory, or story state.

Example with Actions:

Drink the potion | 4 | has_item:Potion | HP+=10;remove_item:Potion

Leaves

Leaves are the choices inside each node. They connect your story flow, handle conditions, and trigger actions. Use the sub-pages for deeper details.

What is a Leaf?

A Leaf is a single option line inside a Node. It's what players can pick. Each Leaf can:

  • Show Text for the choice.
  • Point to another Node (Next).
  • Check Conditions for display.
  • Execute Actions when chosen.

Leaf Structure

Leaves are written as:

Text | Next | Condition | Actions
  • Text ‚Üí What the player sees (supports {var} substitution).
  • Next ‚Üí The Node ID to go to (can be a single ID, random from a list like 1/2/3, or a variable).
  • Condition ‚Üí Optional checks that must be true for the Leaf to be visible.
  • Actions ‚Üí Things that happen when the player chooses this Leaf.

Simple Example:

Take the sword | 5 | has_item:Key | add_item:Sword

Comments

You can add comments to your Leaf options to leave notes for yourself or to temporarily disable a line. The game engine will ignore any line that starts with a hash symbol (#).

Example with comments:

# This is a choice to go left.
    Go left | 2 | | 
    Go right | 3 | | 
    # TODO: Add a condition for this secret path later
    # A secret path | 4 | has_item:key |

Conditions

Conditions control whether a Leaf is visible to the player. You can chain multiple conditions together using & or ;. All conditions must be true for the Leaf to appear.

  • has_item:Sword ‚Üí Checks if the player has the "Sword" item.
  • not_has_item:Key ‚Üí Checks if the player does not have the "Key" item.
  • HP <= 0 ‚Üí Allows for complex mathematical and boolean checks using variables.
  • var:CLICKS==1 ‚Üí An explicit way to check a variable's value (the var: prefix is optional).

Chance-Based Visibility Example:

Search the forest | 8 | chance(40) |  
    Search the cave | 9 | chance(90) |

In this node, "Search the forest" has a 40% chance to be visible, and "Search the cave" has a 90% chance to be visible. It's possible for both, one, or neither to appear. This is different from the weighted() action, which picks one outcome from a pool. You can mix this with other conditions, for example: has_item:torch & chance(50). The leaf will only be visible if the player has a torch AND the 50% chance roll succeeds.

Checking a Header Character (hlet):

This choice is only visible if the second letter of this node's header is 'x' | 10 | hlet:2=x |

The hlet condition (short for "header letter") checks a specific character in the current node's header string. It's case-sensitive. The index is 1-based, not 0-based.

Timed Choice Example (lifetime):

Quick, grab the idol! | 12 | lifetime(10) | add_item:Idol

The lifetime(SECONDS) condition makes a choice available for only a limited time. When the node loads, a 10-second timer starts for this choice. If the player doesn't pick it within that time, the choice will disappear. This can be combined with other conditions, like has_item:torch & lifetime(5).

You can also use ! before any condition to negate its result. This is a powerful shortcut for writing "not" conditions.

Example:

Don't talk to the dragon | 2 | !has_item:Sword |

This choice is only visible if the player does not have the Sword. It works the same as not_has_item:Sword.

You can also use it with other conditions:

The safe is locked... | 5 | !hlet:4=K & !has_item:Key |

This choice will appear if the fourth character of the header is not 'K' AND the player does not have a "Key" in their inventory.

Actions

Actions are commands that execute when a Leaf is chosen. You can chain multiple actions together using & or ;.

Variables

  • gold = 100 ‚Üí Assigns a value to a variable.
  • gold += 5 ‚Üí Modifies a variable's value (also supports -=, *=, /=).

Inventory

  • add_item:Sword ‚Üí Adds an item to the player's inventory.
  • remove_item:Key ‚Üí Removes an item from the player's inventory.
  • clearinv ‚Üí Wipes the entire inventory.
  • rename_item:OLD,NEW ‚Üí Changes the name of an item in the player's inventory from OLD to NEW.

Renaming an Item Example:

You polish the sword until it shines. | 5 | | rename_item:Rusty Sword,Polished Sword

If the player's inventory contains "Rusty Sword" when this option is chosen, its name will be changed to "Polished Sword".

Utility Actions

    ¬†
  • clamp(HP:0,100) ‚Üí Ensures the 'HP' variable stays between 0 and 100. If HP was 110, it becomes 100. If it was -5, it becomes 0. The min/max values can be numbers or other variables.
    • ¬†
  • consume(Key:goto:12) ‚Üí A shortcut action. It first checks if the player has "Key". If so, it removes "Key" from the inventory and then executes the action (goto:12). If the player doesn't have the item, nothing happens.
    • ¬†
  • rlet:INDEX:CHAR ‚Üí A specific action that replaces a character in the current node's header. See below for details.
¬†

Replacing a Header Character (rlet):

¬† 
Swap the second letter to 's' | 5 | | rlet:2:s
¬†

The rlet action (short for "replace letter") changes a specific character in the current node's header string. The index is 1-based, not 0-based.

Flow Control

  • once:gold+=1 ‚Üí Runs this action only once ever.
  • once:>gold+=1 ‚Üí Runs only the first action once.
  • once:>>gold+=1;log:done ‚Üí Runs all actions, but only once total.
  • once:<gold+=1>log:done ‚Üí Conditional gold increase (if condition true), but log always, only the first time.

Randomization

  • randr(damage:5,20) ‚Üí Sets the 'damage' variable to a random number between 5 and 20.
  • rands(weather:Sunny,Cloudy) ‚Üí Sets the 'weather' variable to either "Sunny" or "Cloudy".
  • weighted(weather:Sunny=70,Rain=30) ‚Üí Sets 'weather' based on weights. "Sunny" has a 70% chance, "Rain" has a 30% chance.
  • chance(30)>gold+=10 ‚Üí A 30% chance to execute the success action (gold+=10).
  • chance(30)>gold+=10>gold-=5 ‚Üí A 30% chance for success (gold+=10), otherwise the fail action (gold-=5) is executed.

Loops

  • repeat:3>gold+=1 ‚Üí Repeats the first action (gold+=1) 3 times.
  • repeat:3>>gold+=1;log:done ‚Üí Repeats all actions 3 times.
  • repeat:3:<gold+=1>log:done ‚Üí Adds gold if the condition is true, but always logs.

Instant Actions (@)

Instant actions are special actions that run automatically when a player enters a node, before they get to make a choice. They are defined on their own line, starting with @.

@HP-=5
    @if(is_cursed)>add_item:Curse

Timed Instant Actions (@timer)

A special type of instant action that executes after a delay. Supports the same separators as if and repeat.

@timer(SECONDS):>ACTION
    @timer(SECONDS):>>ACTION1;ACTION2
    @timer(SECONDS):<CACTS>UACTS
  • > ‚Üí Only runs the first action after the timer expires.
  • >> ‚Üí Runs all actions after the timer expires.
  • <...> ‚Üí Conditional actions inside the block, unconditional actions outside it.

Example:

@timer(5):>>goto:10;HP-=5

After 5 seconds, the player is sent to Node 10 and loses 5 HP.

Conditional Actions (if)

The if statement is a powerful action that lets you run other actions only if a certain condition is met. It has a few different forms.

1. Single Conditional Action (>)

This is the simplest form. If the condition is true, it executes only the first action that follows the >.

if(has_item:Sword)>damage+=10;cost=50

In this case, if the player has a sword, damage will increase by 10. The cost=50 action is not part of the if-statement and will always run.

2. Multi-Action Block (>>)

By using >>, you can tell the if-statement to treat all subsequent actions on the line as a single conditional block.

if(gold>=price)>>potions+=1;gold-=price

Here, if gold is greater than or equal to price, both actions (potions+=1 and gold-=price) will be executed.

3. Advanced Conditional/Unconditional Block (<...>)

This is the most flexible syntax. It allows you to define a block of actions that run if the condition is true, followed by actions that run regardless.

if(condition):<conditional_actions>unconditional_actions
if(gold>=price):<potions+=1;gold-=price>price*=2

Let's break it down:

  • Condition: gold>=price
  • Conditional Actions: potions+=1;gold-=price (inside the <...>) will only run if the condition is true.
  • Unconditional Action: price*=2 (after the <...>) will always run, no matter what the condition was.
